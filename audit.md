 ### 🔴 Спецификация vs Backend — naming convention конфликт

 Спецификация явно указывает:

 │ Важно: Все i18n-поля передаются со строчной n: nameI18n, titleI18n, descriptionI18n

 Реальность backend: pydantic.alias_generators.to_camel("title_i18n") = titleI18N (заглавная N).

 ```
   Спецификация:  titleI18n, nameI18n, descriptionI18n, valueI18n
   Backend JSON:  titleI18N, nameI18N, descriptionI18N, valueI18N
 ```

 │ Компонент                                         │ Использует │ Совпадает с backend? │
 ├───────────────────────────────────────────────────┼────────────┼──────────────────────┤
 │ Спецификация (product-creation-flow.md)           │ titleI18n  │ 🔴 НЕТ               │
 ├───────────────────────────────────────────────────┼────────────┼──────────────────────┤

 3. FRONTEND ADMIN ↔ BACKEND

 ### 🔴 CRITICAL: Три backend-маршрута НЕ СУЩЕСТВУЮТ (Media upload flow)

 Frontend admin вызывает через BFF-прокси три URL, которых нет в main backend:

 ┌─────────────────────────────────────────────────────┬────────────────────────────────────────────────────────┬─────────────┐
 │ Frontend вызывает                                   │ Проксирует на backend                                  │ Существует? │
 ├─────────────────────────────────────────────────────┼────────────────────────────────────────────────────────┼─────────────┤
 │ POST /api/catalog/products/{id}/media/upload        │ POST /api/v1/catalog/products/{id}/media/upload        │ 🔴 НЕТ      │
 ├─────────────────────────────────────────────────────┼────────────────────────────────────────────────────────┼─────────────┤
 │ POST /api/catalog/products/{id}/media/{mid}/confirm │ POST /api/v1/catalog/products/{id}/media/{mid}/confirm │ 🔴 НЕТ      │
 ├─────────────────────────────────────────────────────┼────────────────────────────────────────────────────────┼─────────────┤
 │ POST /api/catalog/products/{id}/media/external      │ POST /api/v1/catalog/products/{id}/media/external      │ 🔴 НЕТ      │
 └─────────────────────────────────────────────────────┴────────────────────────────────────────────────────────┴─────────────┘

 Эти операции существуют только в image_backend:
 - POST /api/v1/media/upload → presigned URL
 - POST /api/v1/media/{storage_object_id}/confirm → запуск обработки
 - POST /api/v1/media/external → импорт по URL

 Проблема: Frontend admin проксирует через main backend (Next.js API route → backendFetch()), но main backend не имеет ни этих маршрутов, ни прокси к
 image_backend. ImageBackendClient в backend поддерживает только DELETE.

 Результат: Весь flow загрузки медиа нерабочий. Фронтенд получит 404 на каждой из трёх операций.

 ### 🔴 CRITICAL: Field name mismatches в media upload flow

 Даже если прокси будет исправлен:

 ┌─────────────────────┬─────────────────────────┬──────────────────────────┐
 │ Что                 │ Frontend ожидает        │ Image backend возвращает │
 ├─────────────────────┼─────────────────────────┼──────────────────────────┤
 │ Presigned URL field │ slot.presignedUploadUrl │ presignedUrl             │
 ├─────────────────────┼─────────────────────────┼──────────────────────────┤
 │ Object ID field     │ slot.id                 │ storageObjectId          │
 └─────────────────────┴─────────────────────────┴──────────────────────────┘

 ┌─────────────────────┬─────────────────────────────────────────────┬───────────────────────────┐
 │ Что                 │ Frontend отправляет                         │ Image backend принимает   │
 ├─────────────────────┼─────────────────────────────────────────────┼───────────────────────────┤
 │ Upload request body │ { mediaType, role, contentType, sortOrder } │ { contentType, filename } │
 └─────────────────────┴─────────────────────────────────────────────┴───────────────────────────┘

 mediaType, role, sortOrder — это каталожные концепции, image_backend их не знает.

 ### 🔴 CRITICAL: Отсутствие en locale при создании продукта

 useProductForm.js:323-325:

 ```js
   titleI18N: {
     ru: state.titleRu,
     ...(state.titleEn ? { en: state.titleEn } : {}),  // en пропускается если пусто!
   },
 ```

 Backend I18nDict требует оба locale {ru, en}. Если пользователь не заполнит английское название → backend вернёт 422 MISSING_REQUIRED_LOCALES.

 Аналогично для descriptionI18N (строки 332-334).

 ### 🟠 MAJOR: descriptionI18n default в backend — невалидный

 ProductCreateRequest.description_i18n: I18nDict = Field(default_factory=dict) — если фронтенд не передаёт поле, default = {}, что не пройдёт валидатор
 I18nDict (требует ru + en). Продукт нельзя создать без описания, хотя спека говорит что поле опционально.

 ### 🟠 MAJOR: countryOfOrigin в create-запросе игнорируется

 useProductForm.js:340-342 отправляет countryOfOrigin, но ProductCreateRequest не имеет этого поля (только ProductUpdateRequest). Значение молча
 отбрасывается Pydantic.

 ### 🟠 MAJOR: Нет polling для статуса обработки медиа

 Image backend предоставляет SSE-эндпоинт GET /media/{id}/status и статусы PENDING_UPLOAD → PROCESSING → COMPLETED/FAILED. Frontend admin никогда не
 опрашивает статус обработки — после confirmMedia() сразу переходит к следующему шагу. Если обработка ещё идёт, при привязке к продукту URL/variants будут
 пустыми.

 ### 🟡 MINOR: Frontend не вызывает completeness endpoint

 GET /products/{id}/completeness реализован в backend, но frontend admin не использует его. Вместо этого используется клиентская эвристика isPublishable
 (проверяет только наличие variant attrs + price). missingRequired / missingRecommended нигде не отображаются.

 ### 🟡 MINOR: FSM UI — только DRAFT → ENRICHING

 Frontend admin поддерживает только один переход DRAFT → ENRICHING (кнопка "На модерацию"). Нет UI для:
 - ENRICHING → READY_FOR_REVIEW
 - READY_FOR_REVIEW → PUBLISHED
 - PUBLISHED → ARCHIVED
 - Обратных переходов (ENRICHING → DRAFT, ARCHIVED → DRAFT)

 ### 🟡 MINOR: Нет отправки version при обновлениях

 Frontend admin не отправляет поле version в PATCH-запросах (нет edit page вообще). Optimistic locking работает только на DB-уровне через version_id_col,
 но application-level early check пропускается.

 ### 🔴 CRITICAL: i18n field names не совпадают с backend

 frontend/main/lib/types/catalog.ts:

 ```typescript
   titleI18n: Record<string, string>;    // lowercase 'n'
   descriptionI18n: Record<string, string>;
   nameI18n: Record<string, string>;
 ```

 Backend отдаёт titleI18N, descriptionI18N, nameI18N (uppercase N). При подключении API все i18n поля будут undefined.

 ### 🟠 MAJOR: ProductStatus enum не совпадает

 ```typescript
   // frontend/main/lib/types/catalog.ts:8
   export type ProductStatus = "DRAFT" | "ACTIVE" | "ARCHIVED" | "DELETED";
 ```

 Backend:

 ```python
   # UPPERCASE enum values в ответах (StrEnum lowercase: draft, enriching, ...)
   ProductStatus: draft | enriching | ready_for_review | published | archived
 ```

 Расхождения:
 - "ACTIVE" (frontend) — не существует в backend (нет такого статуса)
 - "DELETED" (frontend) — не существует в backend (soft-delete, не статус)
 - enriching, ready_for_review, published — отсутствуют в frontend type
