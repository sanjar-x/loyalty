## Research Summary

### Libraries & frameworks

- **FastAPI** -- APIRouter with path parameters, request/response models. Existing codebase uses `@inject` + `FromDishka` pattern for DI in route handlers.
- **SQLAlchemy 2.1 (async)** -- `version_id_col` on Product/SKU ORM models already provides optimistic locking. On version mismatch, `StaleDataError` is raised during flush. Repository must catch this and re-raise as a domain `ConcurrencyError`.
- **Dishka DI** -- `Provider` with `Scope.REQUEST`, `CompositeDependencySource = provide(...)` pattern. All repos and handlers registered per-request. Container assembled in `container.py`.
- **attrs** -- Domain entities use `@attr.dataclass` (not stdlib dataclass). `AggregateRoot` mixin provides `add_domain_event()`.
- **Pydantic v2** -- Presentation schemas inherit `CamelModel` (camelCase aliasing). Read models in application layer use `BaseModel`.
- **hashlib (stdlib)** -- SHA-256 for variant hash computation. No external dependency.

### Patterns

- **Data Mapper** -- Repositories have `_to_domain()` and `_to_orm()` methods for bidirectional mapping. ORM models never leak into domain.
- **CQRS** -- Separate `Command` dataclasses + `CommandHandler` classes for writes; `QueryHandler` classes for reads returning read models.
- **Unit of Work** -- All writes via `async with self._uow:` ... `await self._uow.commit()`. Aggregates registered for event extraction.
- **Domain Events** -- Dataclasses inheriting `DomainEvent` with `aggregate_type`/`event_type` overrides. Events are deferred for Product (per enhancement plan), so MVP handlers will NOT emit domain events.

### Decisions

- **Domain events for Product are DEFERRED** -- per enhancement plan, `domain_events_product_lifecycle` is P2. Product command handlers will not emit events in this implementation.
- **Supplier remains FK-only** -- no domain entity needed. Product domain entity holds `supplier_id: uuid.UUID | None`.
- **ProductStatus enum** -- will be defined in domain value_objects.py (not reuse ORM enum) to maintain domain purity. Maps to existing ORM `ProductStatus`.
- **Money value object** -- simple frozen attrs dataclass with `amount: int` and `currency: str`. Validation: amount >= 0, currency is 3-char ISO code.
- **Variant hash** -- computed by Product aggregate's `add_sku` / `update_sku` methods using `hashlib.sha256`.
- **Product list endpoint** -- GET /api/v1/catalog/products with basic pagination (limit/offset) for admin MVP. QA report noted this was missing from PRD endpoint table; we include it.

## Micro-Task 2: Add Product and SKU domain entities

**Layer:** Domain
**Module:** catalog
**Type:** New feature
**FR Reference:** FR-001, FR-002, FR-005, FR-006
**Pattern:** product_aggregate_root, product_status_lifecycle_fsm, optimistic_locking_in_domain, variant_hash_algorithm

**Goal:**
Create the Product aggregate root and SKU child entity in the domain layer. Product owns SKUs, enforces status lifecycle transitions (FSM), carries version for optimistic locking, and computes variant hashes for SKU uniqueness.

**Files to create/modify:**

- `src/modules/catalog/domain/entities.py` -- add Product (AggregateRoot) and SKU child entity classes

**Acceptance criteria:**

- [ ] Product is an attrs dataclass extending AggregateRoot with fields: id, slug, title_i18n, description_i18n, status (ProductStatus), brand_id, primary_category_id, supplier_id (nullable), country_of_origin (nullable), tags, version, deleted_at (nullable), created_at, updated_at, published_at (nullable), skus (list of SKU)
- [ ] Product.create() factory sets status=DRAFT, version=1, empty skus list
- [ ] Product.update() allows updating title, description, slug, brand_id, primary_category_id, supplier_id, country_of_origin, tags
- [ ] Product.soft_delete() sets deleted_at timestamp
- [ ] Product has FSM methods: transition_status(new_status) that validates allowed transitions per the defined table (Draft->Enriching, Enriching->Draft, Enriching->ReadyForReview, ReadyForReview->Enriching, ReadyForReview->Published, Published->Archived, Archived->Draft). Sets published_at on transition to PUBLISHED.
- [ ] Product.add_sku() creates a SKU child, computes variant_hash via SHA-256, checks uniqueness across existing skus, raises DuplicateVariantCombinationError on collision
- [ ] SKU is an attrs dataclass (not AggregateRoot) with fields: id, product_id, sku_code, variant_hash, price (Money), compare_at_price (Money, nullable), is_active, version, deleted_at (nullable), created_at, updated_at, variant_attributes (list of tuples: attribute_id, attribute_value_id)
- [ ] SKU validates compare_at_price > price when both are set
- [ ] All existing tests pass after this change
- [ ] Linter/type-checker passes

**Architecture constraints:**

- Product is the aggregate root; SKU is a child entity (not AggregateRoot)
- hashlib.sha256 is stdlib -- allowed in domain
- No domain events emitted (deferred to future phase)
- One aggregate per transaction

**Context7 references:**

- attrs dataclass pattern consistent with existing Brand, Category entities

**Depends on:** MT-1
