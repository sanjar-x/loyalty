## 1. Context & purpose

### What we are building

We are building a Supplier module that gives the platform a structured way to manage where products come from. The platform aggregates streetwear and fashion products from two kinds of sources: Chinese marketplaces (Poizon, Taobao, Pinduoduo, 1688) and local Russian regional suppliers. Today, supplier information lives as a basic database record inside the catalog system with no management screens, no activation controls, and no proper separation. This module extracts supplier management into its own dedicated area of the system, adds the ability to activate and deactivate suppliers, seeds the four known Chinese marketplaces as built-in entries, and connects supplier data cleanly to product records so managers can track where every product comes from.

### Problem statement

The platform currently has no way to manage suppliers as independent business entities. The supplier record exists only as a bare-bones database entry buried inside the catalog system -- there are no screens to create, edit, or deactivate suppliers, no way to distinguish marketplace suppliers from local ones in business workflows, and critical information like the marketplace source link is missing from product management screens. This means managers cannot efficiently track product sourcing, the system cannot enforce rules like "products from Chinese marketplaces must have a source link," and there is no foundation for delivery estimate badges that customers need to see.

### Who this is for

**The manager** -- an internal staff member who creates and manages the supplier directory. They add local suppliers, browse the supplier list when creating products, select the right supplier for each product, and use source links to navigate to Chinese marketplace listings for purchasing.

**The admin** -- a system administrator who oversees all suppliers including the built-in Chinese marketplace entries. They can activate or deactivate any supplier, manage system-seeded marketplace records, and control which suppliers are available for product assignment.

**The customer** -- an end user shopping on the platform who sees delivery information derived from the supplier type (for example, a "from China" badge or an "in stock locally" indicator on product cards) but never interacts with supplier data directly.

## 2. Scope

### What is included (MVP)

- **Supplier as a standalone business entity** -- a proper supplier record with name, type (cross-border marketplace or local), region, and active/inactive status, managed independently from the product catalog (P0)
- **Dedicated supplier module** -- supplier management extracted into its own area of the system with its own business rules, data access, and management screens, following the same organizational pattern as other platform modules (P0)
- **Supplier activation and deactivation** -- ability to mark suppliers as active or inactive; inactive suppliers keep their existing product connections but cannot be assigned to new products (P0)
- **Supplier type classification** -- clear distinction between cross-border marketplace suppliers and local regional suppliers, driving validation rules and delivery information (P0)

### What is included with modifications

- **Marketplace supplier seeding** -- the four known Chinese marketplaces (Poizon, Taobao, Pinduoduo, 1688) are pre-loaded as built-in supplier entries with fixed identifiers, using supplier module constants referenced by the database migration rather than raw database commands. This keeps seed data alongside business logic. (P1)
- **Separated read and write operations** -- supplier creation, updates, and deactivation go through one path; supplier lookups and listings go through another, optimized path. Simplified from the standard approach: a single read interface with filter support covers all lookup needs since the supplier entity is straightforward. (P1)

### What is explicitly NOT included

**Deferred -- will be added later:**

- **Supplier list filtering by type and status** -- query parameters like type, active status, and name search on the supplier list. Activate immediately after basic supplier management is working.
- **Supplier info on product cards** -- showing supplier name and type in product responses for delivery badge rendering. Activate when the frontend needs delivery badges on product cards.
- **URL-friendly supplier identifiers (slugs)** -- a human-readable, unique slug for each supplier. Activate when supplier detail pages or external API consumers need stable business identifiers.
- **Structured contact information** -- dedicated fields for email, phone, and address on suppliers. Activate when local supplier management requires operational contact tracking for order coordination.
- **Extensible metadata field** -- a flexible data field on suppliers for ad-hoc attributes like logo, notes, or delivery estimates. Activate when different supplier types need different custom attributes.
- **Supplier lifecycle state machine** -- multi-step activation workflow (pending, active, blocked) with transition rules. Activate when supplier onboarding requires approval workflows.

**Excluded -- not planned:**

- **Vendor as a text label on products** -- storing supplier as a plain text name on the product instead of a proper linked record. This cannot support structured supplier management, type classification, or independent lifecycle. Identified as an anti-pattern in industry analysis.
- **Cross-module entity sharing** -- importing supplier business objects directly into the catalog system's core logic. This creates brittle dependencies where changes to supplier structure break catalog functionality. The platform uses identifier-based references with a validation interface instead.
- **Separate link table for product-supplier connections** -- a dedicated association table between products and suppliers. The platform has a simple one-to-one relationship (each product comes from exactly one supplier), so a direct reference on the product record is sufficient. Adding a separate link table is unnecessary complexity.
- **Hard deletion of suppliers** -- permanently removing supplier records from the system. This causes data loss or breaks product references. Deactivation preserves historical data while preventing new assignments.

## 3. User scenarios

### Manager: Adding a local supplier and creating a product from it

**Happy path:**

1. Manager opens the supplier management area and sees the list of existing suppliers, including the four built-in Chinese marketplaces
2. Manager clicks "Create Supplier" and enters the name, selects "Local" as the type, and specifies the region (e.g., "Moscow")
3. System creates the supplier and shows it in the list as active
4. Manager navigates to product creation, selects the new local supplier from a dropdown
5. Manager fills in product details; since this is a local supplier, the source link field is optional
6. System saves the product with the supplier connection recorded

**Alternative scenarios:**

- Manager creates a product from a cross-border marketplace supplier (e.g., Poizon) and must provide a valid source link (e.g., https://www.poizon.com/product/12345) -- the system requires it for marketplace suppliers
- Manager updates an existing local supplier's name or region to correct a typo

**Edge cases:**

- What happens when a manager tries to create a product using a deactivated supplier? The system rejects the request with a clear message that the supplier is inactive.
- What happens when a manager searches for a supplier that does not exist by the given identifier? The system returns a "supplier not found" error.

### Admin: Managing marketplace suppliers and deactivating a local supplier

**Happy path:**

1. Admin opens the supplier management area and sees all suppliers including the four built-in marketplace entries
2. Admin selects a local supplier that is no longer in use and clicks "Deactivate"
3. System marks the supplier as inactive; existing products linked to this supplier remain visible and unchanged
4. Admin verifies that the deactivated supplier no longer appears in the product creation supplier dropdown

**Alternative scenarios:**

- Admin updates the name of a marketplace supplier to correct the display name
- Admin reactivates a previously deactivated supplier when the business relationship resumes

**Edge cases:**

- What happens when an admin tries to delete a marketplace supplier? The system does not support deletion -- only deactivation. The marketplace entries persist as system reference data.
- What happens when there are products referencing a supplier being deactivated? The products remain visible and functional; only new product assignments are blocked.

### Customer: Seeing delivery information based on supplier type

**Happy path:**

1. Customer browses the product catalog and sees product cards
2. Products from Chinese marketplace suppliers display a "from China" delivery indicator
3. Products from local suppliers display an "in stock" or local delivery indicator
4. Customer uses this information to set delivery expectations before purchasing

**Alternative scenarios:**

- Customer views a product whose supplier has been deactivated -- the product and its delivery badge remain visible since the product was already created

**Edge cases:**

- What happens when a product's supplier type changes? Supplier type is immutable after creation (a marketplace does not become a local supplier), so this scenario does not arise.

## 4. Functional requirements

### P0 -- Must have

#### FR-001: Supplier as a standalone business entity

- **What:** Create a Supplier entity with core fields: unique identifier (time-sortable UUID), name, type (cross-border or local), region, active status, and timestamps. The entity is intentionally lean -- no bloated optional fields.
- **Why:** Five out of six major e-commerce platforms (Medusa.js, Vendure, Spree, commercetools, Flxpoint) model suppliers as first-class entities rather than text labels. A structured supplier record enables type classification, lifecycle management, and reliable product-supplier connections. Answers: "What fields does Supplier need?" and "Should Supplier be standalone?"
- **Acceptance criteria:**
  - When a new supplier is created with name "Test Supplier", type LOCAL, and region "Moscow", the system stores a record with a generated UUIDv7 identifier, is_active defaults to true, and created_at is set to current timestamp
  - When a supplier is retrieved by identifier, the response includes all core fields: id, name, type, region, is_active, created_at, updated_at
  - When a supplier is created without a name, the system returns a validation error indicating name is required
  - When a supplier is created with an invalid type value (not CROSS_BORDER or LOCAL), the system returns a validation error
- **Industry reference:** Medusa.js (Vendor entity), Vendure (Seller entity), Spree Commerce (Vendor model), commercetools (Channel entity), Flxpoint (Supplier entity)

#### FR-002: Dedicated supplier module with full layer separation

- **What:** Extract supplier management into a dedicated module (src/modules/supplier/) with proper separation of business rules, data access, and management endpoints, following the same organizational pattern used by existing platform modules (catalog, identity).
- **Why:** DDD best practice supported by the Grzybek reference architecture and Medusa.js v2. Supplier has an independent lifecycle (create, update, activate, deactivate), its own business rules (seeding, type classification), and is referenced by other modules. Clean separation prevents supplier changes from breaking catalog functionality. Answers: "Should the Supplier module be fully standalone?"
- **Acceptance criteria:**
  - Supplier business rules (entity definition, validation rules, error types) contain zero references to infrastructure or database concerns
  - The supplier module exposes a public query interface that other modules can use to check supplier existence and active status without accessing supplier internals
  - Supplier management endpoints are accessible at /api/v1/suppliers (not nested under /api/v1/catalog)
  - Creating, reading, updating, and deactivating suppliers works end-to-end through the new module structure
- **Industry reference:** Modular Monolith with DDD (Grzybek reference architecture), Medusa.js v2 (custom Marketplace Module)

#### FR-003: Supplier activation and deactivation

- **What:** Support activating and deactivating suppliers via dedicated endpoints. Deactivated suppliers retain all existing product connections. The system prevents assigning new products to inactive suppliers.
- **Why:** Four industry sources confirm this as the standard supplier lifecycle pattern. Deactivation (not deletion) preserves historical data integrity -- products that were sourced from a supplier maintain that reference even after the supplier is deactivated. Answers: "How should soft-delete work for suppliers?"
- **Acceptance criteria:**
  - When PATCH /api/v1/suppliers/{id}/deactivate is called for an active supplier, the supplier's is_active becomes false and the response confirms the change
  - When PATCH /api/v1/suppliers/{id}/activate is called for an inactive supplier, the supplier's is_active becomes true
  - When a product creation request references an inactive supplier (is_active=false), the system returns a 422 error with message indicating the supplier is inactive
  - When a supplier is deactivated, all existing products referencing that supplier remain unchanged and accessible
  - When attempting to deactivate a supplier that is already inactive, the system returns a 409 conflict or succeeds idempotently
- **Industry reference:** Saleor (is_private flag), Spree Commerce (state machine), commercetools (version-controlled updates), soft-delete best practices

#### FR-004: Supplier type classification

- **What:** Classify each supplier as either CROSS_BORDER (Chinese marketplace) or LOCAL (Russian regional supplier) using a type field. This classification drives validation rules (source link requirements), delivery information (customer-facing badges), and operational workflows.
- **Why:** Aggregator platforms universally classify suppliers by geography and logistics model. The type drives different business rules: cross-border products must have marketplace source links; local products have different delivery expectations. The SupplierType value already exists in the catalog system and needs to be moved to the supplier module. Answers: "What validation rules should CreateProductCommand enforce?"
- **Acceptance criteria:**
  - When creating a supplier with type CROSS_BORDER, the supplier record stores the type and it is returned in all supplier responses
  - When creating a supplier with type LOCAL, the supplier record stores the type and it is returned in all supplier responses
  - When a product is created referencing a CROSS_BORDER supplier without providing a source_url, the system returns a validation error: "source_url is required for cross-border suppliers"
  - When a product is created referencing a LOCAL supplier without a source_url, the system accepts the request (source_url is optional for local suppliers)
  - The supplier type cannot be changed after creation (type is immutable)
- **Industry reference:** E-commerce aggregator platforms, cross-border e-commerce architecture guides, Poizon/Dewu reseller integration pattern

### P1 -- Should have

#### FR-005: Source link on product for marketplace purchasing

- **What:** Surface the source_url field (link to the original marketplace listing) across all layers of the product entity -- domain model, creation commands, API requests, and API responses. For cross-border suppliers, source_url is required and must be a valid URL. For local suppliers, it is optional.
- **Why:** Three sources confirm source_url as the standard way aggregators link local products to external marketplace listings. The field already exists in the database but is missing from the product domain model, creation commands, and API schemas -- a documented technical debt item. Managers use this link daily to navigate to Poizon, Taobao, etc. to purchase products. Answers: "Should source_url live on Product?" and "What validation rules apply?"
- **Acceptance criteria:**
  - When creating a product with supplier type CROSS_BORDER and source_url "https://www.poizon.com/product/12345", the system stores the URL and returns it in the product response as sourceUrl (camelCase)
  - When creating a product with supplier type CROSS_BORDER and source_url "not-a-url", the system returns a validation error indicating the URL format is invalid
  - When creating a product with supplier type CROSS_BORDER and no source_url, the system returns a validation error: "source_url is required for cross-border suppliers"
  - When creating a product with supplier type LOCAL and no source_url, the system accepts the request and stores source_url as null
  - When retrieving a product, source_url is included in the response (as sourceUrl in camelCase)
- **Industry reference:** Poizon/Dewu reseller pattern, Flxpoint (source_sku/source_url), e-commerce aggregator architecture

#### FR-006: Cross-module product-supplier connection with validation

- **What:** Products reference suppliers via a unique identifier stored on the product record, with database-level referential integrity. When creating or updating products, the system validates that the referenced supplier exists and is active, using a shared query interface between the catalog and supplier modules. The supplier_id on products becomes mandatory (not optional).
- **Why:** Three authoritative sources support this pragmatic approach for a platform sharing a single database. The supplier identifier already exists on products but is currently optional; the business intent is that every product must have a supplier. Application-layer validation via a shared interface keeps modules cleanly separated while the database constraint ensures data integrity. Answers: "How should the cross-module FK relationship work?" and "How should the Supplier module expose its data to catalog?"
- **Acceptance criteria:**
  - When creating a product with a valid, active supplier_id, the system accepts the request and stores the supplier reference
  - When creating a product with a supplier_id that does not exist, the system returns a 404 error: "Supplier not found"
  - When creating a product without a supplier_id, the system returns a validation error: "supplier_id is required"
  - When creating a product with a supplier_id referencing an inactive supplier, the system returns a 422 error: "Supplier is inactive"
  - The supplier module's query interface is the only way the catalog module accesses supplier data -- no direct imports of supplier internal objects
- **Industry reference:** Modular Monolith with DDD (Grzybek), Medusa.js v2 (module links), Vendure (channel assignment)

#### FR-007: Pre-loaded marketplace suppliers

- **What:** The four known Chinese marketplace suppliers (Poizon, Taobao, Pinduoduo, 1688) are seeded as built-in entries during database migration. Each has a fixed, deterministic identifier so references are consistent across all environments (development, staging, production). These entries are created as CROSS_BORDER type, active by default, and editable only by admins (no deletion). Seed data is defined as constants in the supplier module's business logic layer, called from the database migration script.
- **Why:** Three sources confirm that marketplace suppliers are system reference data, not user-created entities. Fixed identifiers enable deterministic references in tests and across environments. Keeping seed constants in the business logic layer (rather than raw database commands) ensures the data definitions stay close to the business rules they serve. Answers: "How should marketplace suppliers be seeded?" and "Should marketplaces be immutable?"
- **Acceptance criteria:**
  - After running the database migration, exactly four suppliers exist: Poizon (type=CROSS_BORDER, region="China"), Taobao (type=CROSS_BORDER, region="China"), Pinduoduo (type=CROSS_BORDER, region="China"), 1688 (type=CROSS_BORDER, region="China")
  - All four marketplace suppliers have fixed, documented UUIDv7 identifiers that are the same across all environments
  - Running the migration multiple times does not create duplicate entries (idempotent)
  - All four marketplace suppliers are created with is_active=true
  - Admins can update the name or region of marketplace suppliers but cannot delete them
- **Industry reference:** Poizon/Dewu integration pattern, Alembic seed data patterns, e-commerce aggregator architecture

#### FR-008: Read and write operation separation for supplier module

- **What:** Supplier write operations (create, update, deactivate, activate) are handled through dedicated command objects and handlers. Read operations (get by ID, list all) are handled through a separate read-optimized path with dedicated data transfer objects. A simplified single read interface with filter methods covers all read patterns.
- **Why:** The platform architecture mandates separated read and write paths across all modules. Existing modules (catalog, identity) already follow this pattern. For the supplier entity -- a single, straightforward aggregate -- a single read interface is sufficient rather than one handler per query type. Answers: "How should the Supplier module expose its data?"
- **Acceptance criteria:**
  - Creating a supplier goes through a CreateSupplierCommand processed by a command handler
  - Updating a supplier goes through an UpdateSupplierCommand processed by a command handler
  - Deactivating a supplier goes through a DeactivateSupplierCommand processed by a command handler
  - Listing suppliers returns read-optimized data transfer objects (not full domain entities)
  - The read interface supports retrieving a single supplier by ID and listing all suppliers
  - Cross-module supplier lookups (from catalog) use a dedicated query interface, not the write-side repository
- **Industry reference:** DDD Aggregate Root Pattern (industry standard), CQRS Read/Write Separation (industry standard)

## 5. Data model

### Entities

#### `Supplier`

- **Purpose:** Represents a product source -- either a Chinese marketplace or a local/regional company. This is the central entity of the supplier module.
- **Key attributes:**
  - `id` -- UUID (v7, time-sortable) -- primary key, auto-generated
  - `name` -- string (max 255) -- display name of the supplier, required
  - `type` -- SupplierType enum (CROSS_BORDER, LOCAL) -- immutable after creation
  - `region` -- string (max 100) -- geographic region (e.g., "China", "Moscow"), required
  - `is_active` -- boolean -- default true, controls whether new products can reference this supplier
  - `version` -- integer -- optimistic locking counter, starts at 1
  - `created_at` -- timestamp -- set on creation
  - `updated_at` -- timestamp -- updated on every modification
- **Relationships:**
  - Has many Products (one-to-many via supplier_id FK on Product)
- **Design rationale:** Five industry sources confirm supplier as a first-class entity with minimal core fields. The lean design avoids bloated nullable columns; extensibility can be added later via a metadata field if needed.

#### `SupplierType`

- **Purpose:** Value object that classifies suppliers into two categories that drive different business rules and customer experiences.
- **Key attributes:**
  - `CROSS_BORDER` -- Chinese marketplace suppliers (Poizon, Taobao, Pinduoduo, 1688); products require source_url; customers see "from China" delivery badge
  - `LOCAL` -- Russian regional suppliers; source_url is optional; customers see "in stock" / local delivery badge
- **Design rationale:** Three industry sources confirm geographic/logistics classification as standard for aggregator platforms. This enum exists today in the catalog module and will be moved to the supplier module.

#### `Product` (cross-module, modified)

- **Purpose:** Existing catalog entity that gains a required supplier connection and a source_url field surfaced across all layers.
- **Key attributes (modified):**
  - `supplier_id` -- UUID FK to Supplier -- changes from optional to required
  - `source_url` -- string (max 1024), nullable -- link to external marketplace listing; required when supplier type is CROSS_BORDER; already exists in database but added to domain entity, commands, and API schemas
- **Relationships:**
  - Belongs to Supplier (many-to-one)
- **Design rationale:** Three sources confirm source_url on Product as the standard aggregator pattern for 1:1 product-supplier relationships. Making supplier_id required aligns the data model with business intent (every product must come from somewhere).

#### `ISupplierQueryService`

- **Purpose:** Interface that the catalog module uses to validate supplier references without depending on supplier internals.
- **Key attributes:**
  - `get_supplier_info(supplier_id)` -- returns supplier ID, name, type, and is_active status
  - `assert_supplier_active(supplier_id)` -- raises error if supplier does not exist or is inactive
- **Relationships:**
  - Implemented by the supplier module's infrastructure layer
  - Consumed by the catalog module's product creation handler
- **Design rationale:** Three authoritative DDD sources recommend shared query interfaces for cross-module validation in a modular monolith with a shared database.

#### `SupplierReadModel`

- **Purpose:** Read-optimized data transfer object for supplier list and detail views.
- **Key attributes:**
  - `id` -- UUID
  - `name` -- string
  - `type` -- string (CROSS_BORDER or LOCAL)
  - `region` -- string
  - `is_active` -- boolean
  - `created_at` -- timestamp
- **Design rationale:** CQRS pattern requires separate read models from domain entities. The read model is a flat, serializable object optimized for API responses.

### Entity relationship summary

A Supplier has a type classification (SupplierType: CROSS_BORDER or LOCAL) that is set at creation and never changes. Each Product belongs to exactly one Supplier via a required supplier_id reference. When a Supplier is deactivated (is_active=false), its existing Products remain linked and visible, but no new Products can be assigned to it. The catalog module validates supplier references through the ISupplierQueryService interface, never accessing supplier internals directly.

### Migration notes

The Supplier ORM model currently exists inside the catalog module (catalog/infrastructure/models.py) with fields: id, name, type, region, created_at, updated_at. Migration involves:

1. Adding `is_active` column (boolean, default true) to the existing suppliers table
2. Adding `version` column (integer, default 1) for optimistic locking
3. Moving the ORM model class from catalog module to supplier module (physical table name stays the same)
4. Moving the SupplierType enum from catalog/domain/value_objects.py to supplier module
5. Surfacing source_url in the Product domain entity, commands, and schemas (column already exists in database)
6. Evaluating whether supplier_id can be made NOT NULL (requires data audit for existing NULL values)
7. Seeding four marketplace suppliers (Poizon, Taobao, Pinduoduo, 1688) with fixed UUIDv7 identifiers

```json
  "name": "My Local Supplier",

  "region": "Moscow"

```

**Supplier response:**

```json
{
  "id": "01912345-6789-7abc-...",
  "name": "My Local Supplier",
  "type": "LOCAL",
  "region": "Moscow",
  "isActive": true,
  "createdAt": "2026-03-24T12:00:00Z",

```

**Pagination:** Supplier list uses the existing PaginatedResponse pattern with page/size parameters and total count in response metadata.

**API naming:** All responses use camelCase field names (e.g., isActive, createdAt, sourceUrl) per platform convention.

Supplier management screens (information architecture, not UI design):

- **Supplier detail view** -- all supplier fields, list of linked products count, edit and deactivate actions
- **Deactivate confirmation** -- shows count of products linked to the supplier, requires confirmation

## 7. Non-functional requirements

### Performance

- API response for single supplier retrieval: p95 < 100ms
- API response for supplier list (up to 100 entries): p95 < 200ms
- Supplier validation during product creation (ISupplierQueryService call): p95 < 50ms
- Database query for supplier by ID: < 10ms (single row by primary key)

### Security

- **Authentication:** All supplier endpoints require a valid authenticated session
- **Authorization:**
  - Manager role: can create local suppliers, list suppliers, view supplier details
  - Admin role: can perform all operations including deactivation, activation, and editing marketplace suppliers
  - Customer role: no direct access to supplier endpoints; supplier-derived information (delivery badges) exposed only through product responses
- **Validation:** Name length 1-255 characters; region length 1-100 characters; type must be a valid enum value; supplier_id must be a valid UUID format
- **Audit:** All supplier create, update, activate, and deactivate operations are logged with actor identity and timestamp

### Scalability

- Expected supplier volume: fewer than 100 active suppliers (4 marketplaces + local suppliers). The entity is low-cardinality.
- Expected product volume per supplier: up to 10,000 products per marketplace supplier, up to 500 per local supplier
- Concurrency: optimistic locking via version field prevents lost updates when multiple admins edit the same supplier simultaneously

### Compliance

Not applicable. No specific regulatory or compliance standards apply to supplier entity management.

| 4   | How should the Dishka DI container wire the ISupplierQueryService for cross-module dependency? | Catalog module needs supplier module's implementation resolved at startup. Module initialization order matters.                               | No -- solvable with standard DI configuration   |

### Risks

| Risk                                                                                       | Type          | Severity | Mitigation                                                                                                                                                 |
| ------------------------------------------------------------------------------------------ | ------------- | -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Migrating Supplier ORM from catalog to supplier module may break FK constraints or imports | Technical     | Medium   | Keep physical table name unchanged. Move only ORM class and domain code. Test FK constraint from products table remains valid.                             |
| Making supplier_id NOT NULL on Product may fail if NULL values exist                       | Technical     | Medium   | Audit existing data first. If NULLs exist, assign to a default supplier or use application-layer validation as a transitional step.                        |

## 10. Roadmap

### MVP (P0)

Implementation order respects dependencies:

1. **Supplier module structure** (supplier_standalone_module) -- create src/modules/supplier/ with domain, application, infrastructure, and presentation layers. No dependencies.
2. **Supplier type classification** (supplier_type_classification) -- move SupplierType enum from catalog to supplier module. Depends on #1.
3. **Supplier entity with core fields** (supplier_as_separate_entity) -- create Supplier domain entity, ORM model with is_active, repository interface, and CRUD endpoints. Depends on #1.
4. **Supplier activation/deactivation** (is_active_deactivation) -- add activate/deactivate commands and endpoints. Depends on #3.

**Definition of done:**

- All six supplier CRUD endpoints return correct responses
- Supplier domain entity passes unit tests for creation, update, and deactivation rules
- Supplier ORM model has is_active and version columns via Alembic migration
- Deactivated supplier correctly blocks new product assignment
- All new code follows existing module patterns (attrs entities, Pydantic schemas, Dishka DI)

### v1.1

- **Source link on products** (source_url_on_product) -- surface source_url in Product domain, commands, and API schemas with cross-border validation
- **Cross-module supplier validation** (cross_module_uuid_fk) -- implement ISupplierQueryService, make supplier_id required on products
- **Marketplace supplier seeding** (marketplace_seed_migration) -- seed Poizon, Taobao, Pinduoduo, 1688 via Alembic migration with fixed UUIDs
- **Read/write separation** (cqrs_supplier_module) -- implement command handlers, query handlers, and read models

### Future
