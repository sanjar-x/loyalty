  ★ Insight ─────────────────────────────────────
  DC-1 — это самая распространённая проблема в DDD-системах с CQRS: create() и update() идут через разные code paths, и инварианты забывают продублировать.
  Правильное решение — вынести i18n dict в self-validating Value Object (class I18nText), который в конструкторе всегда проверяет completeness. Тогда
  невалидное значение невозможно создать ни через create, ни через update, ни через прямое присвоение.

  DC-5 — filter_settings как opaque JSON — это осознанный trade-off "frontend owns the schema". Но без хотя бы JSON Schema валидации backend не может
  гарантировать, что storefront projections не упадут на malformed filter_settings. В идеале — хотя бы max_depth + max_keys проверка (которую
  BoundedJsonDict уже даёт на уровне schema).

  1. Data Timeliness — Данные актуальны (cache freshness, sync lag)

  Оценка: 7.5/10

  Кеширование в подсистеме атрибутов

  Подсистема атрибутов не кеширует сами атрибуты/значения, но кеширует производные данные — storefront projections и effective attribute sets:

  ┌────────────────────────────────────────┬───────────────┬───────────────────────────────────────┐
  │               Cache key                │      TTL      │            Что кешируется             │
  ├────────────────────────────────────────┼───────────────┼───────────────────────────────────────┤
  │ catalog:template:{id}:effective_attrs  │ 3600s (1 час) │ Resolved attribute set для template   │
  ├────────────────────────────────────────┼───────────────┼───────────────────────────────────────┤
  │ catalog:storefront:filters:{cat_id}    │ 3600s         │ Filterable attributes для категории   │
  ├────────────────────────────────────────┼───────────────┼───────────────────────────────────────┤
  │ catalog:storefront:card:{cat_id}       │ 3600s         │ Card-visible attributes для категории │
  ├────────────────────────────────────────┼───────────────┼───────────────────────────────────────┤
  │ catalog:storefront:comparison:{cat_id} │ 3600s         │ Comparable attributes для категории   │
  ├────────────────────────────────────────┼───────────────┼───────────────────────────────────────┤
  │ catalog:storefront:form:{cat_id}       │ 3600s         │ Form attributes для категории         │
  ├────────────────────────────────────────┼───────────────┼───────────────────────────────────────┤
  │ catalog:category_tree                  │ 300s (5 мин)  │ Дерево категорий с template_id        │
  └────────────────────────────────────────┴───────────────┴───────────────────────────────────────┘

  Explicit invalidation (активная инвалидация)

  Функция invalidate_template_effective_cache() (resolve_template_attributes.py:110-137):
  1. Удаляет template effective attrs cache
  2. Находит все категории с этим effective_template_id
  3. Удаляет 4 storefront caches для каждой категории
  4. Вызывается из: bind_attribute_to_template, unbind_attribute_from_template, update_template_attribute_binding, reorder_template_bindings,
  set_attribute_value_active

  ★ Insight ─────────────────────────────────────
  Ключевая проблема (DT-1..DT-5): Cache invalidation покрывает только binding-level изменения (привязка/отвязка attribute к template, изменение
  requirement_level, reorder, activate/deactivate value). Но attribute-level и value-level изменения (переименование атрибута, изменение ui_type,
  добавление/удаление/обновление значений) не инвалидируют cache.

  Это создаёт "инвалидационный зазор": если PM переименовал атрибут "Цвет" в "Оттенок" (update_attribute), storefront filters будет показывать "Цвет" ещё
  час. Если дизайнер добавил новое значение "Бирюзовый" (add_attribute_value), оно не появится в фильтрах до TTL expiry.

  Архитектурная причина: invalidate_template_effective_cache() привязана к template-binding lifecycle, а не к attribute/value lifecycle. Нужен аналогичный
  invalidate_attribute_dependents_cache(), который по attribute_id находит все templates через get_template_ids_for_attribute(), а затем инвалидирует кеши
  связанных категорий.

  ★ Insight ─────────────────────────────────────
  TR-6 (AttributeValue без timestamps) — это значимый пробел. В e-commerce каталоге attribute values — это "живые" данные: PM добавляет новые цвета,
  деактивирует устаревшие размеры, переименовывает материалы. Без created_at/updated_at невозможно:
  - Построить audit trail ("когда добавили размер XXS?")
  - Отладить проблему ("когда это значение стало неактивным?")
  - Реализовать incremental sync (CDC) в поисковый индекс

  TR-2..TR-4 (cascade без per-child events) — это фундаментальный trade-off DDD: aggregate root "владеет" дочерними сущностями, и при удалении root дочерние
   сущности удаляются как часть единой операции. Но для event-driven систем это создаёт "shadow deletions" — downstream consumers узнают только о смерти
  root, но не о смерти каждого child. Решение: либо emit batch-event с deleted_value_ids: list[UUID], либо emit per-child events в loop перед cascade
  delete.

  ★ Insight ─────────────────────────────────────
  MS-1 + MS-2 — это самый серьёзный findings во всём анализе. Проект находится в состоянии "migration vacuum": ORM описывает одну schema, а DB (если создана
   из последней миграции) — другую. Это типичная ситуация при крупном рефакторинге (families → templates), когда "потом напишем миграцию" превращается в
  "migration debt".

  Последствия для production deploy:
  1. alembic upgrade head упадёт — нет файлов
  2. Base.metadata.create_all создаст clean DB с текущей schema, но потеряет все данные
  3. Ручной ALTER TABLE возможен, но без alembic revision — нет истории, нет downgrade

  Правильный путь: написать одну squashed миграцию, которая:
  1. Переименует таблицы (ALTER TABLE attribute_families RENAME TO attribute_templates)
  2. Переименует FK columns
  3. Добавит новые columns (is_active, timestamps где нужны)
  4. Удалит устаревшие columns/tables (family_attribute_exclusions, is_visible_in_catalog)
  5. Обновит enum types если нужно
  6. Содержит полноценный downgrade()

  MS-4 (тесты bypass Alembic) — это анти-паттерн: тесты не валидируют что миграции корректны. В production-ready проекте тесты должны использовать alembic
  upgrade head для setup, чтобы ловить schema drift.
