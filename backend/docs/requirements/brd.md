## 1. Executive Summary

Модуль «Корзина покупок» (Cart) — ключевой компонент гибридной e-commerce платформы, работающей по модели агрегатора-дропшиппера. Платформа не имеет собственного склада: товар закупается у поставщиков только после оплаты покупателем. Основной канал продаж — Telegram Mini App с его ограничениями по UI и стабильности сети.

Цель модуля — обеспечить покупателю удобный и прозрачный процесс формирования заказа: от добавления товара до перехода к оплате. Ключевая бизнес-проблема — покупатель работает с товарами из двух принципиально разных источников (cross-border из Китая и локальные поставщики), которые отличаются сроками доставки, условиями возврата и ценовыми рисками. Корзина должна явно разделять эти группы, обеспечивая прозрачность условий покупки.

Ожидаемый результат — рост конверсии из каталога в оформленный заказ за счёт понятного UX, минимального трения при checkout и доверия покупателя к прозрачности условий. Фазовый roadmap предусматривает четыре этапа: от MVP с базовой функциональностью до масштабирования с промо-механиками и аналитикой.

## 2. Project Overview / Background

### 2.1 Бизнес-контекст

Платформа работает как гибридный агрегатор-дропшиппер (аналог Farfetch). Бизнес-модель:

- Товар не хранится на складе — закупается у поставщика после оплаты покупателем
- Два типа источников: **CROSS_BORDER** (Китай: Poizon, Taobao, Pinduoduo, 1688) и **LOCAL** (локальные поставщики)
- Доход формируется как наценка к закупочной цене
- Каноническая валюта — RUB, все суммы хранятся в минимальных единицах (копейки) через Money Value Object
- Конвертация CNY → RUB выполняется на этапе создания SKU в Catalog BC, не в Cart

### 2.2 Текущая ситуация (As-Is)

Модуль корзины отсутствует. Покупатель может просматривать каталог, но не может сформировать заказ. Весь путь от выбора товара до покупки — разорван. Это блокирует запуск модулей Order, Payment и Notification.

### 2.3 Недостатки текущей ситуации

- Невозможность совершить покупку через платформу
- Отсутствие воронки конверсии «каталог → корзина → заказ → оплата»
- Блокировка разработки зависимых модулей (Order, Payment)
- Невозможность сбора аналитики по поведению покупателей

### 2.4 Бизнес-драйверы

1. **Монетизация** — без корзины и checkout платформа не генерирует выручку
2. **Конкурентное давление** — аналогичные платформы (Poizon App, KazanExpress) уже предоставляют полный цикл покупки
3. **Разблокировка pipeline** — Order, Payment, Notification зависят от Cart
4. **Аналитика** — данные корзины критичны для понимания спроса и оптимизации ассортимента

## 3. Business Objectives

### Objective 1: Запуск полного цикла покупки

| Параметр      | Значение                                                             |
| ------------- | -------------------------------------------------------------------- |
| Objective     | Обеспечить покупателю возможность собрать и оформить заказ           |
| Key Result 1  | 100% покупателей могут добавить товар в корзину и перейти к checkout |
| Key Result 2  | Конверсия «каталог → корзина» ≥ 8% в первый месяц после запуска      |
| Key Result 3  | Конверсия «корзина → оформленный заказ» ≥ 40%                        |
| Target Metric | Количество оформленных заказов > 0 в первую неделю                   |

### Objective 2: Прозрачность условий покупки

| Параметр      | Значение                                                                 |
| ------------- | ------------------------------------------------------------------------ |
| Objective     | Покупатель понимает сроки, условия доставки и возврата до оформления     |
| Key Result 1  | Товары разделены по источникам с явным указанием сроков                  |
| Key Result 2  | Показатель «неожиданных» обращений в поддержку по срокам < 5% от заказов |
| Key Result 3  | NPS по прозрачности условий ≥ 7/10                                       |
| Target Metric | Доля отменённых заказов по причине «не знал о сроках» < 3%               |

### Objective 3: Минимизация потерь конверсии

| Параметр      | Значение                                                                     |
| ------------- | ---------------------------------------------------------------------------- |
| Objective     | Сократить потери покупателей на этапе корзины из-за технических и UX-проблем |
| Key Result 1  | Время загрузки корзины < 2 секунд (p95)                                      |
| Key Result 2  | Abandoned cart rate < 65% (среднерыночный — 70%)                             |
| Key Result 3  | Успешный merge гостевой корзины при авторизации в 99%+ случаев               |
| Target Metric | Drop-off на этапе checkout < 30%                                             |

## 8. Business Rules

| ID       | Описание                               | Условие                           | Действие                                                      |
| -------- | -------------------------------------- | --------------------------------- | ------------------------------------------------------------- |
| RULE-001 | Единственный источник цены — Catalog   | Цена в корзине всегда = SKU.price | При каждом отображении корзины актуализировать цену           |
| RULE-002 | Группировка по типу источника          | SKU связан с Supplier             | CROSS_BORDER  LOCAL                                           |
| RULE-003 | Ограничение количества на позицию      | qty > 0 и qty ≤ MAX_QTY_PER_ITEM  | Попытка выставить qty за пределами — отклоняется с сообщением |
| RULE-004 | Фиксация цены при checkout initiate    | Покупатель нажимает «Оформить»    | Создаётся price snapshot с TTL (начальное значение: 15 минут) |
| RULE-005 | Валидация перед confirm                | Покупатель нажимает «Подтвердить» | Проверка: все SKU is_active=true, цены совпадают со snapshot; |
| RULE-006 | Пересчёт при истечении TTL snapshot    | TTL snapshot истёк                | Автоматический пересчёт; уведомление покупателя               |
| RULE-007 | Merge-стратегия гостевой корзины       | гостевой корзиной                 | Merge: дубликаты — qty суммируется;                           |
| RULE-008 | Валюта — только RUB                    | Все расчёты в корзине             | Суммы в (копейки);                                            |
| RULE-009 | Пустая корзина — empty state           | Корзина не содержит позиций       | Отображается empty state с CTA «Перейти в каталог»            |
| RULE-010 | Деактивированный SKU в корзине         | SKU.is_active = false             | Позиция помечается как недоступная;                           |
| RULE-011 | Условия возврата привязаны к источнику | SupplierType = CROSS_BORDER       | Возврат только при браке; информация отображается в корзине   |
| RULE-012 | Условия возврата для LOCAL             | SupplierType = LOCAL              | Стандартный возврат 7 дней; информация отображается в корзине |
| RULE-013 | Минимальный tap-target в Telegram      | Все  элементы в корзине           | Минимальный размер touch-target: 44×44 px                     |

## 9. Assumptions & Constraints

### 9.1 Допущения (Assumptions)

| ID   | Допущение                                                                                                          |
| ---- | ------------------------------------------------------------------------------------------------------------------ |
| A-01 | Модуль Catalog предоставляет актуальную цену SKU через синхронный Query Service с приемлемой латентностью (<200ms) |
| A-02 | Конвертация CNY → RUB выполнена на этапе создания SKU и не требует повторного расчёта в Cart                       |
| A-03 | Customer ID однозначно идентифицируется через JWT или Telegram initData                                            |
| A-04 | Telegram Mini App поддерживает localStorage для гостевой корзины                                                   |
| A-05 | Покупатель имеет доступ к списку ПВЗ через модуль Geo без дополнительной интеграции                                |
| A-06 | Модуль Order будет принимать данные checkout через Domain Event или Command                                        |
| A-07 | Одна корзина на одного покупателя (не multi-cart)                                                                  |

### 9.2 Ограничения (Constraints)

| ID   | Ограничение                                                                                                    |
| ---- | -------------------------------------------------------------------------------------------------------------- |
| C-01 | **Telegram WebView**: нет вкладок, ограниченный viewport, может быть свёрнуто OS, MainButton/BackButton API    |
| C-02 | **Нет резервирования инвентаря**: дропшиппинг-модель не предполагает резерва — товар может стать недоступен    |
| C-03 | **Курсовые колебания**: цена CROSS_BORDER товаров может измениться из-за курса CNY/RUB (фиксируется в Catalog) |
| C-04 | **DDD-архитектура**: Cart — отдельный Bounded Context; общение с другими модулями только через контракты       |
| C-05 | **Стек**: Python/SQLAlchemy, PostgreSQL — без выбора альтернативных технологий                                 |
| C-06 | **Touch-first**: все UI-элементы ≥ 44×44 px, оптимизация для мобильных экранов                                 |
| C-07 | **Money VO**: все денежные операции через Value Object Money (amount в копейках, currency=RUB)                 |

## 14. Glossary

| Термин               | Определение                                                                                   |
| -------------------- | --------------------------------------------------------------------------------------------- |
| SKU                  | Stock Keeping Unit — минимальная единица товара (конкретный размер/цвет варианта)             |
| Bounded Context (BC) | Ограниченный контекст в DDD — автономный модуль с собственной доменной моделью                |
| CROSS_BORDER         | Тип поставщика: товары из-за рубежа (Китай), доставка 10-25 дней                              |
| LOCAL                | Тип поставщика: локальные поставщики, доставка 2-7 дней                                       |
| Money VO             | Value Object для денежных сумм: amount (в минимальных единицах) + currency                    |
| Price Snapshot       | Фиксация цен на момент initiate checkout с ограниченным TTL                                   |
| Merge (корзин)       | Объединение гостевой (localStorage) и серверной корзины при авторизации                       |
| ПВЗ                  | Пункт выдачи заказов — точка получения товара покупателем                                     |
| TTL                  | Time To Live — время жизни объекта (в данном контексте — price snapshot)                      |
| Abandoned Cart       | «Брошенная» корзина — корзина с товарами без перехода к checkout в течение 24+ часов          |
| MoSCoW               | Метод приоритизации: Must / Should / Could / Won't                                            |
| initData             | Данные аутентификации Telegram Mini App                                                       |
| Дропшиппинг          | Модель продаж без собственного склада: товар закупается у поставщика после оплаты покупателем |
| Empty State          | UI-состояние для пустой корзины с призывом к действию                                         |
