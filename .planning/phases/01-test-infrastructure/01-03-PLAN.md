    **File 1: `backend/tests/utils/query_counter.py`** (per D-09)

    Build `assert_query_count` context manager using SQLAlchemy's `after_cursor_execute` event:

    ```python
    from __future__ import annotations
    from contextlib import contextmanager
    from sqlalchemy import event
    from sqlalchemy.ext.asyncio import AsyncSession

    @contextmanager
    def assert_query_count(session: AsyncSession, expected: int, *, label: str = ""):
        """Context manager that counts SQL queries and asserts exact count.

        Hooks into SQLAlchemy's after_cursor_execute event on the underlying
        sync connection. Filters out SAVEPOINT and RELEASE SAVEPOINT statements
        since those are artifacts of the nested-transaction test isolation pattern.

        Args:
            session: The async session to monitor.
            expected: Expected number of application-level SQL queries.
            label: Optional label for the assertion error message.

        Usage:
            with assert_query_count(db_session, expected=1, label="list_brands"):
                result = await brand_repo.list_all()
        """
    ```

    CRITICAL implementation detail: In the project's async SQLAlchemy setup with nested transactions, the event listener MUST be attached to the sync connection obtained from the async session. The approach:

    1. Get the sync session: `sync_session = session.sync_session`
    2. Get the underlying sync connection: `sync_conn = sync_session.connection()`
    3. Attach `after_cursor_execute` event listener to `sync_conn`
    4. In the listener callback, increment counter but SKIP statements that start with `SAVEPOINT` or `RELEASE SAVEPOINT` or `ROLLBACK TO SAVEPOINT` (these are test isolation artifacts, not application queries)
    5. On exit, remove the listener and assert `count == expected`

    HOWEVER: Getting the sync connection from an async session in a nested-transaction context is tricky. The executor should try these approaches in order:
    - Approach A: `session.sync_session.connection()` -- may not work if no active transaction
    - Approach B: Listen on the engine level via `session.get_bind()` -- but may catch queries from other sessions
    - Approach C: Use `@event.listens_for(session.sync_session, "after_cursor_execute")` -- session-level event

    The implementer MUST test the chosen approach against the actual `db_session` fixture from `tests/conftest.py` to ensure queries are counted correctly. Write the validation test (`test_query_counter_smoke.py`) FIRST to verify the approach works.

    Also build an alternative version that works as an async context manager if the sync approach proves problematic:
    ```python
    @asynccontextmanager
    async def assert_query_count_async(session, expected, label=""):
        ...
    ```

    ```python
    import pytest
    from sqlalchemy import text
    from tests.utils.query_counter import assert_query_count

    @pytest.mark.integration
    class TestQueryCounter:
        async def test_counts_single_query(self, db_session):
            """Verify counter detects exactly 1 query."""
            with assert_query_count(db_session, expected=1, label="single_select"):
                await db_session.execute(text("SELECT 1"))

        async def test_counts_multiple_queries(self, db_session):
            """Verify counter detects exactly N queries."""
            with assert_query_count(db_session, expected=3, label="three_selects"):
                await db_session.execute(text("SELECT 1"))
                await db_session.execute(text("SELECT 2"))
                await db_session.execute(text("SELECT 3"))

        async def test_fails_on_wrong_count(self, db_session):
            """Verify counter raises AssertionError on mismatch."""
            with pytest.raises(AssertionError, match="Expected 5 queries, got 1"):
                with assert_query_count(db_session, expected=5):
                    await db_session.execute(text("SELECT 1"))

        async def test_excludes_savepoint_statements(self, db_session):
            """Verify SAVEPOINT/RELEASE are not counted."""
            # The db_session fixture already uses begin_nested() which creates
            # savepoints. This test verifies those don't inflate the count.
            with assert_query_count(db_session, expected=1, label="savepoint_exclusion"):
                await db_session.execute(text("SELECT 1"))
    ```
