## 1. Зачем мы это делаем

Мы — агрегатор, который объединяет товары с Poizon, Taobao, Pinduoduo, 1688 и локальных поставщиков. У нас нет своего склада. Наша ценность — витрина, удобный выбор и доставка до ПВЗ. Модель как у Farfetch.

Проблема: товары от разных источников приходят с разными описаниями. У одного поставщика "Цвет" — это текстовое поле, у другого — код из палитры, у третьего вообще нет. Покупатель не может нормально фильтровать, сравнивать, выбирать.

**Нам нужна единая система атрибутов каталога**, которая позволит:

- Описывать любой товар одним языком, независимо от источника
- Привязывать нужные характеристики к категориям — с указанием, какие обязательны, какие нет
- Поддерживать мультиязычность с первого дня (узбекский, русский, английский — минимум)
- Дать фронтенду понятные инструкции: где показать фильтр, где — цветной кружок, где — ползунок
- Управлять поведением атрибутов на витрине: фильтрация, поиск, сравнение, видимость
- Контролировать полноту заполнения карточки товара (completeness)
- Обеспечить совместимость с внешними каналами продаж (маркетплейсы, Google Shopping)

### Чему нас научили лидеры рынка

Изучив как устроены каталоги у Magento/Adobe Commerce, Shopify, Akeneo PIM, Amazon, commercetools и Microsoft Dynamics Commerce, мы выделили ключевые паттерны, которые нужно заложить с первого дня:

**Magento** разделяет значения атрибутов по типам данных в отдельные хранилища (varchar, int, decimal, text, datetime). У них есть «Attribute Sets» (наборы атрибутов для типа товара) и «Attribute Groups» (визуальные секции в админке). Каждый атрибут несёт флаги поведения: фильтруемый, поисковый, сравниваемый, видимый на витрине, вес в поиске.

**Shopify** пошёл по пути стандартной таксономии: к каждой категории привязаны предопределённые атрибуты с готовыми значениями. Акцент на кросс-канальной совместимости — атрибуты маппятся на Google, Facebook, Amazon.

**Akeneo PIM** ввёл понятие «Family» (семейство товаров) с наследованием атрибутов по иерархии. Атрибуты могут быть обязательными по-разному для разных каналов. Есть расчёт completeness — насколько заполнена карточка.

**Amazon** требует при привязке к категории разделение атрибутов на обязательные и опциональные, а также выделяет «variation themes» — атрибуты, по которым создаются варианты товара.

**commercetools** разделяет атрибуты на уровень продукта и уровень варианта, с ограничениями: «уникально для каждого варианта», «одинаково для всех вариантов».

**Мы берём лучшее из каждого подхода, но не копируем сложность Magento EAV целиком.**

## 2. Кто пользователи

| Кто | Что делает с атрибутами |
|-----|------------------------|
| **Контент-менеджер** | Создаёт атрибуты и значения, организует их в группы, привязывает к категориям, задаёт обязательность и поведение на витрине |

| **Фронтенд** | Получает данные об атрибутах с типом отображения, флагами поведения, значениями — строит UI |
| **Каналы продаж** | Маркетплейсы и Google Shopping получают атрибуты в своих форматах (на будущее) |

## 3. Что такое «Атрибут» в нашей системе

Атрибут — это характеристика товара, которую мы определяем один раз и потом используем в любых категориях.

### 3.1 Основные свойства атрибута

- **Уникальный код** — машинное имя, не меняется (например, `color`, `sole_type`, `screen_size`)
- **Человекочитаемый slug** — для URL-ов
- **Название** — на всех поддерживаемых языках
- **Описание** — на всех поддерживаемых языках (для подсказок в UI)
- **Тип данных** — что за значение хранится: строка, число, десятичное, булево, дата
- **Тип отображения** — как фронтенд должен это показать: кнопки-чипы, выпадающий список, цветные кружки, ползунок диапазона и т.п.
- **Признак «справочный»** — есть ли у атрибута фиксированный список допустимых значений (да/нет)

### 3.2 Группа атрибутов (новая сущность)

**Зачем:** и Magento, и Akeneo, и Dynamics 365 группируют атрибуты визуально — чтобы в админке и на карточке товара они не шли сплошным списком, а были разбиты по смыслу.

Группа атрибутов — это логическая секция для отображения в админке и на витрине.

Примеры групп:
- «Основные» — название, бренд, страна
- «Физические характеристики» — размер, вес, цвет, материал
- «Технические» — разрешение экрана, объём батареи
- «Маркетинговые» — описание, теги, SEO-текст
- «Логистические» — тип упаковки, вес посылки

Каждая группа имеет:
- **Код** — уникальный
- **Название** — мультиязычное
- **Порядок сортировки** — в каком порядке показывать секции

Каждый атрибут принадлежит ровно одной группе. Группа по умолчанию — «Общие».

### 3.3 Флаги поведения атрибута на витрине (новое)

**Зачем:** Magento, Dynamics 365, Google Retail, commercetools — все управляют поведением атрибута отдельными флагами. Без этого контент-менеджер не может сказать системе «этот атрибут показывай в фильтрах, но не используй для поиска».

Каждый атрибут несёт следующие флаги:

| Флаг | Что значит | Пример |
|------|-----------|--------|
| **Фильтруемый** | Атрибут доступен как фильтр на витрине (layered navigation) | Цвет — да. Инструкция по уходу — нет |
| **Поисковый** | Значение атрибута участвует в полнотекстовом поиске | Название бренда — да. Вес — нет |
| **Вес в поиске** | Приоритет при ранжировании поисковых результатов (1–10) | Бренд = 8, описание = 3 |
| **Сравниваемый** | Атрибут участвует в таблице сравнения товаров | Размер экрана — да. SEO-текст — нет |
| **Видимый на карточке** | Показывать на странице товара | Материал — да. Внутренний код — нет |
| **Видимый в каталоге** | Показывать в превью товара (карточка в листинге) | Цена — да. Описание — нет |

По умолчанию все флаги выключены. Контент-менеджер включает нужные для каждого атрибута.

### 3.4 Правила валидации (новое)

**Зачем:** Akeneo и commercetools позволяют задавать ограничения на значения. Это ловит ошибки при заполнении, до того как плохие данные попадут на витрину.

Для каждого атрибута можно задать правила валидации в зависимости от типа данных:

| Тип данных | Доступные правила |
|-----------|-------------------|
| Строка | Минимальная/максимальная длина, regex-шаблон (email, URL, etc.) |
| Число / Десятичное | Минимум, максимум, допустимость отрицательных |
| Дата | Минимальная / максимальная дата |

Правила валидации — опциональные. Если не заданы, ограничений нет.

## 4. Что такое «Значение атрибута»

Если атрибут справочный (например, «Цвет» со списком: Красный, Синий, Чёрный), то у него есть набор значений. Каждое значение имеет:

- **Код** — уникальный в рамках атрибута (`red`, `blue`, `black`)
- **Slug** — для URL
- **Название** — мультиязычное
- **Алиасы для поиска** — список синонимов, чтобы поиск по «алый» нашёл «Красный»
- **Метаданные** — произвольная доп. информация (например, HEX-код цвета `#FF0000` для отображения кружка, или URL иконки)
- **Группа** — опциональная группировка значений (например, «Тёплые тона» / «Холодные тона»)
- **Порядок сортировки** — в каком порядке показывать в UI

### 4.1 Маппинг на внешние системы (новое, подготовка)

**Зачем:** Shopify выстраивает всю таксономию вокруг совместимости с Google, Facebook, Amazon. Нам предстоит работать с маркетплейсами — нужно заложить фундамент.

Каждое значение может нести дополнительные внешние коды в метаданных:

```
meta_data: {
  "hex": "#FF0000",
  "google_color": "Red",
  "amazon_color_map": "Red"
}
```

На этом этапе это просто соглашение о структуре метаданных. Формальный маппинг — отдельный модуль в будущем.

## 5. Что такое «Правило привязки атрибута к категории»

Не все атрибуты нужны всем категориям. «Размер подошвы» не нужен в «Сумках», а «Тип застёжки» не нужен в «Кроссовках».

### 5.1 Базовая связь (как было)

Правило — это связь: **такой-то атрибут используется в такой-то категории**, с указанием порядка отображения. Одна пара «категория + атрибут» может существовать только один раз.

### 5.2 Обязательность атрибута в категории (новое)

**Зачем:** и Magento, и Amazon, и Akeneo требуют указания обязательности. Без этого невозможно рассчитать completeness товара и контролировать качество каталога.

При привязке атрибута к категории указывается:
- **Обязательный** — товар в этой категории обязан иметь значение для этого атрибута. Без него карточка считается неполной.
- **Рекомендуемый** — атрибут желателен, влияет на completeness, но не блокирует публикацию.
- **Опциональный** — атрибут доступен, но не влияет на completeness.

По умолчанию — опциональный.

### 5.3 Переопределение флагов поведения на уровне категории (новое)

**Зачем:** Dynamics 365 Commerce позволяет переопределять поведение атрибута (filterable, searchable) на уровне канала. Нам нужна такая же гибкость на уровне категории.

Атрибут «Материал» глобально может быть нефильтруемым, но в категории «Сумки» контент-менеджер хочет включить фильтр по материалу (кожа / текстиль / замша).

Поэтому при привязке атрибута к категории можно переопределить любой флаг поведения (раздел 3.3). Если не переопределён — используется глобальное значение из атрибута.

### 5.4 Настройка фильтра на уровне категории (новое)

**Зачем:** Dynamics 365 позволяет задавать тип отображения фильтра (список, ползунок, диапазон) и пороги. Это нужно для разных категорий.

Для фильтруемых атрибутов в конкретной категории можно указать:
- **Тип фильтра:** список значений, диапазон (от–до), ползунок
- **Пороги диапазона** (для числовых): например, для цены — `0; 50000; 100000; 500000; 1000000`

Если не указано — фронтенд использует тип отображения из глобальных настроек атрибута.

## 6. Полнота карточки товара — Completeness (новое)

**Зачем:** это один из ключевых паттернов Akeneo. Completeness показывает, насколько заполнена карточка товара, и позволяет контролировать качество каталога без ручной проверки каждой карточки.

### 6.1 Как это работает

Для каждого товара в каждой категории система может рассчитать процент заполненности:

```
Completeness = (заполненные обязательные атрибуты + заполненные рекомендуемые × 0.5) 
               / (все обязательные + все рекомендуемые × 0.5) × 100%
```

Формула — ориентировочная, точную согласовать с PM.

### 6.2 Зачем бизнесу

- Контент-менеджер видит дашборд: «в категории Кроссовки 340 товаров заполнены на 100%, 58 товаров — меньше 70%»
- Можно ставить правило: «не публиковать товар, если completeness ниже 80%»
- Можно приоритизировать работу по наполнению каталога

### 6.3 Что входит в этот этап

На этом этапе мы **не строим дашборд** и **не блокируем публикацию**. Мы закладываем данные, по которым completeness можно рассчитать: обязательность атрибута в привязке к категории. Сам расчёт и UI — следующий этап.

## 7. Уровень атрибута: продукт vs. вариант (новое, подготовка)

**Зачем:** commercetools, Magento, Akeneo — все разделяют атрибуты на уровень продукта и уровень варианта (SKU). Это критически важно для работы с вариантами.

Примеры:
- **Уровень продукта** (одинаково для всех вариантов): бренд, страна производства, описание, инструкция по уходу
- **Уровень варианта** (разное для каждого SKU): цвет, размер, цена, вес, штрихкод

### Что делаем на этом этапе

Добавляем к атрибуту поле «уровень»: **продукт** или **вариант** (по умолчанию — продукт).

Это чисто информационное поле — пока нет модуля товаров, оно не влияет на логику. Но когда модуль появится, эта разметка уже будет готова, и не придётся мигрировать атрибуты.

## 8. Сценарии использования

### 8.1 Управление группами атрибутов (новое)

**Создание группы**
Контент-менеджер создаёт группу: код, название на нужных языках, порядок.

**Обновление/удаление группы**
Удалить группу можно только если в ней нет атрибутов. При удалении атрибуты не удаляются — они переносятся в группу «Общие».

### 8.2 Управление атрибутами

**Создание атрибута**
Указывает код, slug, название, тип данных, тип отображения, является ли справочным, группу, флаги поведения, правила валидации, уровень (продукт/вариант). Система проверяет уникальность кода и slug.

**Обновление атрибута**
Можно изменить: название, описание, тип отображения, группу, флаги поведения, правила валидации, уровень (с ограничениями при наличии товаров). Нельзя менять: код, slug, тип данных (если атрибут используется).

**Удаление атрибута**
Только если не привязан ни к одной категории.

**Просмотр атрибутов**
Список с пагинацией, поиском по названию, фильтрацией по: типу данных, типу UI, справочности, группе, уровню (продукт/вариант), флагам (фильтруемый, поисковый и т.д.).

### 8.3 Управление значениями атрибутов

Без изменений от v1 (добавление, обновление, удаление, массовая сортировка, просмотр).

### 8.4 Управление привязками атрибутов к категориям (расширено)

**Привязка атрибута к категории**
Контент-менеджер выбирает категорию, атрибут, указывает:
- Порядок отображения
- Обязательность (обязательный / рекомендуемый / опциональный)
- Переопределения флагов поведения (необязательно)
- Настройки фильтра (необязательно)

**Отвязка, массовая сортировка, просмотр** — без изменений.

**Массовое обновление обязательности** (новое)
Контент-менеджер может за один запрос изменить обязательность нескольких атрибутов в категории.

### 8.5 Получение данных для витрины (расширено)

**Атрибуты категории для фильтров**
Фронтенд запрашивает: «Дай мне все фильтруемые атрибуты категории "Кроссовки" с их значениями». Получает упорядоченный список, где для каждого атрибута указан: тип фильтра, значения с переводами, метаданные, пороги диапазона (если числовой).

**Атрибуты категории для карточки товара**
Фронтенд запрашивает: «Дай мне все видимые-на-карточке атрибуты категории "Кроссовки"». Получает атрибуты, сгруппированные по группам атрибутов, с указанием обязательности.

**Атрибуты категории для сравнения**
Фронтенд запрашивает: «Дай мне все сравниваемые атрибуты категории "Кроссовки"». Получает список для таблицы сравнения.

**Атрибуты категории для формы создания товара**
Админка запрашивает: «Дай мне все атрибуты категории "Кроссовки" для заполнения карточки». Получает полный набор с группами, обязательностью, типами виджетов, правилами валидации.

**Атрибут по slug** — без изменений.

## 9. Бизнес-правила (сводка)

### Атрибуты
1. Код атрибута — глобально уникальный, неизменяемый после создания
2. Slug атрибута — глобально уникальный, неизменяемый после создания
3. Название обязательно хотя бы на одном языке
4. Нельзя удалить атрибут, если он привязан хотя бы к одной категории
5. Изменение типа данных запрещено, если атрибут уже используется (на будущее)
6. Каждый атрибут принадлежит ровно одной группе атрибутов
7. Вес в поиске — от 1 до 10, по умолчанию 5

### Группы атрибутов
8. Код группы — глобально уникальный, неизменяемый
9. Нельзя удалить группу, если в ней есть атрибуты (либо автоматический перенос в «Общие»)
10. Группа «Общие» (код: `general`) существует всегда и не может быть удалена

### Значения атрибутов
11. Код значения уникален в рамках одного атрибута
12. Slug значения уникален в рамках одного атрибута
13. Значения можно добавлять только к справочным атрибутам
14. Порядок сортировки значений управляется вручную

### Привязки к категориям
15. Одну пару «категория + атрибут» можно создать только один раз
16. Порядок атрибутов внутри категории управляется вручную
17. Обязательность по умолчанию — «опциональный»
18. Переопределения флагов — опциональны; если не заданы, используются глобальные значения атрибута

### Валидация
19. Правила валидации проверяются при сохранении значения атрибута в товаре (на будущее)
20. Правила валидации должны соответствовать типу данных атрибута (нельзя задать min/max для строки)

## 10. Что мы хотим получить на выходе

### 10.1 API-эндпоинты

**Группы атрибутов (новое):**
- Создать группу
- Получить группу по ID
- Получить список групп (с пагинацией)
- Обновить группу
- Удалить группу

**Атрибуты:**
- Создать атрибут
- Получить атрибут по ID
- Получить атрибут по slug
- Получить список атрибутов (с фильтрами и пагинацией)
- Обновить атрибут
- Удалить атрибут

**Значения атрибутов:**
- Добавить значение к атрибуту
- Обновить значение
- Удалить значение
- Получить значения атрибута (с пагинацией и поиском)
- Массовая пересортировка значений

**Привязки атрибутов к категориям:**
- Привязать атрибут к категории (с обязательностью, переопределениями, настройками фильтра)

- Отвязать атрибут от категории
- Массовая пересортировка привязок
- Массовое обновление обязательности

**Витринные эндпоинты (расширены):**
- Получить фильтруемые атрибуты категории (с значениями)

- Получить атрибуты для формы создания товара (полный набор с валидацией)

## 11. Мультиязычность

Все текстовые поля, видимые конечному пользователю (название атрибута, описание, название группы, название значения), хранятся на всех поддерживаемых языках.

При поиске по названию система ищет по всем языкам одновременно. Поиск по алиасам — тоже по всем языкам.

---

## 12. Что НЕ входит в этот этап

- Модуль товаров (Product) и привязка конкретных значений к товарам
- Валидация «атрибут используется в товарах» при удалении
- **Наследование атрибутов по дереву категорий** — Akeneo поддерживает до 20 уровней, нам это может быть полезно, но это отдельная задача. На этом этапе привязки делаются напрямую к конкретной категории.
- Completeness: расчёт, дашборд, блокировка публикации (закладываем данные — обязательность атрибута, но не строим расчёт)
- Flat-index / денормализованное представление для витрины (Magento использует это для ускорения — решим при оптимизации)
- Импорт/маппинг атрибутов из внешних источников
- Формальный маппинг на внешние таксономии (Google, Amazon)
- Публичный API для внешних интеграторов
- Ролевая модель и разграничение доступа

## 13. Критерии приёмки

1. Все эндпоинты из раздела 10.1 работают, задокументированы (OpenAPI/Swagger)
2. Все бизнес-правила из раздела 9 соблюдаются и проверены тестами
3. Мультиязычность: можно создать атрибут с названием на 3 языках, найти его поиском по любому
4. Массовые операции работают атомарно: либо все позиции обновились, либо ни одна
5. API возвращает понятные ошибки с кодами
6. Витринный эндпоинт «фильтруемые атрибуты категории» возвращает данные, достаточные для построения фильтров
7. Витринный эндпоинт «атрибуты для формы товара» возвращает данные, сгруппированные по группам, с обязательностью и валидацией
8. Привязка атрибута к категории поддерживает обязательность и переопределение флагов
9. События генерируются для каждого ключевого действия
10. Код следует архитектурным соглашениям проекта

---

## 14. Приоритет

**Это фундамент каталога.** Без атрибутов мы не можем: нормально отображать товары, давать фильтры покупателям, маппить данные от поставщиков, строить сравнение товаров, контролировать качество каталога.

Рекомендуемый порядок реализации:

1. **Группы атрибутов** — лёгкая сущность, нужна до атрибутов
2. **Атрибуты (CRUD + флаги + валидация)** — ядро всей системы
3. **Значения атрибутов** — наполнение справочников
4. **Привязки к категориям (с обязательностью и переопределениями)** — связь с деревом каталога
5. **Витринные эндпоинты** — фронтенд начинает работать
